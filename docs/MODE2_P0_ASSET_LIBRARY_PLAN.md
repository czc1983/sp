# Mode2 P0/P1/P2 资产库、分镜与生成计划

日期：2026-07-19

## 总目标

本轮要把 Mode2 从“能出规则草案”推进到“可持续迭代的新路线骨架”：一键分析原视频后，同时得到资产库、视觉时间线、4-15 秒分镜窗口，并为后续 Seedance2.0 重生成做好目标资产绑定。

Mode2 的最终方向是：原视频作为动作骨架，Seedance2.0 重新生成英语版、欧洲风格、外国人物的新片。资产库必须先解决“谁是谁、哪个图能替换、哪个图只是证据”的问题，否则后面分镜和生成都会漂。

优先级分三层：

1. P0：先把角色身份、SAM3 彩色 mask 候选、身份绑定、分轨反馈和资产分层做好。
2. P1：再把场景/物品资产从“语义猜测”升级成“多帧视觉择优 + 检测/分割/提纯”。
3. P2：最后把分镜管理页做成可操作的横向时间轴、当前镜头资产面板和 Seedance 生成准备层。

## 指挥中心规则

## 竞品实现逻辑（录像观察 + 推断）

我们没有竞品后端源码，只能根据 `C:\Users\Administrator\Desktop\5655.mkv` 的界面行为推断。可以确定的是，它的产品逻辑不是“用户先手工点每个人”，而是先自动分析，再让用户确认资产。

观察到的流程：

1. 原片输入：用户上传一个视频，系统先分析整条原片。
2. 资产详情：自动列出角色列表、场景列表。角色卡通常有原片证据图，也有用户上传或系统生成的目标角色图。
3. 分镜管理：左侧看视频，中间编辑分镜提示词，下方是横向时间轴，右侧显示当前镜头相关角色和场景。
4. 视频编辑：按单个分镜生成，提示词里用资产标签绑定角色和场景，例如 `@角色A`、`@S1`。

推断它大概率做了这些事：

1. 全片理解：用视频理解模型先抽出人物、场景、剧情段落和镜头关系。
2. 视觉聚类：把同一角色在不同时间出现的脸、半身、全身候选聚合到一个角色资产里。
3. 场景聚类：把背景/空间相近的镜头聚合成场景资产，尽量不用人物近景当干净场景图。
4. 资产标签化：给每个角色/场景生成稳定 alias，再在分镜提示词里引用。
5. 人工确认前置：用户主要做“换图、改名、确认资产”，而不是每张图上点身份。

更正后的关键理解：

竞品看起来是“点一次分析原视频，分析完成后分镜和资产库同时可用”。这说明它不是先把分镜完全切好，再让用户慢慢补资产；更像是先做一层全片视觉/语义分析，把角色、场景、镜头边界、资产证据一起建出来，然后才把结果展示成“资产详情”和“分镜管理”两个页面。

所以我们的 Mode2 P0 也要改成这个顺序：

1. 先粗采样/粗切：只为拿到候选帧、运动变化、镜头变化，不把它当最终分镜。
2. 再跑人物/场景识别：SAM3 或缓存 SAM3 彩色 mask 参与识别“有几个人、人物何时出现/消失、人物区域怎么变化”。
3. 再生成资产库：角色证据、场景证据、物品证据先成库，并标出置信度和冲突。
4. 再生成最终分镜：分镜边界参考视觉切点、人物进出、SAM3 区域变化、语义剧情变化，同时满足 Seedance 4-15 秒限制。
5. 最后做资产绑定：每个分镜引用已经确认或待确认的资产 ID。

一句话：SAM3 应该走在“最终分镜展示”前面，但它仍需要粗采样/粗切提供候选帧；不能完全零切分就跑整片。

可借鉴的地方：

1. 资产库先分角色/场景，再进入分镜。
2. 原片证据图和目标替换图必须分开显示。
3. 每个资产要知道会影响哪些分镜。
4. 分镜页只显示当前镜头相关资产，不把全库塞进去。
5. 用户可以改角色名、换目标图、确认资产，然后再生成。

不能照搬的地方：

1. 我们不能假设 Seedance2.0 会理解彩色 mask。当前结论是：mask 不作为 Seedance 生成输入。
2. SAM3 只用于原片识别、身份核对、选帧和分镜绑定辅助。
3. Mode2 不能依赖 Mode1 的转绘链路；只能借鉴函数思路，代码和路由要独立。

## P0 核心原则

1. 原片证据和目标替换图彻底分离。
2. 资产库先纠错再展示，不把明显错误的图当成可用资产摆出来。
3. 一个画面有多个人时，允许在同一张原图/彩色 mask 上绑定多个角色。
4. SAM3 分轨是“角色证据增强”和“镜头绑定辅助”，不是 Seedance 的必需输入。
5. 服务器关闭时不能无限等待，必须显示“只能用缓存/当前不可跑”。
6. 旧项目只当参考，不能触发昂贵重跑，除非用户明确开启。
7. P0 的按钮语义要从“生成分镜草案”升级为“分析原视频”：一次分析产出资产库、视觉时间线、分镜草案，而不是分开猜。

## P0 新的一键分析流水线

入口按钮：`分析原视频并生成资产/分镜`

后端执行顺序：

1. 项目与缓存检查：确认当前视频 hash、旧项目参考策略、是否允许全片理解、是否允许远程 SAM3。
2. 粗视觉时间线：用 PySceneDetect、YOLO、人脸/人物数、运动变化抽候选帧和粗切点。
3. SAM3 身份候选层：对候选关键帧生成或读取彩色 mask，得到每个 person object 的颜色、bbox、面积、中心点和时间。
4. 人物轨迹/身份聚类：把同一颜色/同一空间轨迹/同一人脸特征聚合成角色候选；冲突时进入待确认。
5. 场景/物品候选层：从候选帧里提取场景和物品证据；人物占比高的场景先降级为镜头参考。
6. 语义理解层：把原片粗时间线、关键帧、旧项目故事参考交给视频理解/LLM，输出角色名、关系、剧情段落、场景名。
7. 融合层：视觉结果为主，语义结果为线索；把角色名/场景名贴到视觉资产上。
8. 分镜窗口层：以视觉切点、人物进出、动作变化、语义剧情边界共同决定 4-15 秒 Seedance 分镜。
9. 资产绑定层：每个分镜写入 `role_asset_ids`、`scene_asset_ids`、`prop_asset_ids` 和 `asset_refs`。
10. 展示层：只有经过体检后的资产进入可用区；冲突和低置信资产进入待核对区。

这条流水线的重点是：先得到“可信资产和可信时间线”，再给用户看。用户看到的不是原始乱候选，而是已经被系统筛过的一版资产库。

## P0 数据结构

每个 role asset 至少要区分这些层：

```json
{
  "id": "role_xxx",
  "kind": "role",
  "name": "女主",
  "alias": "@role_hero",
  "source_identity": {
    "status": "candidate|annotated|tracked|needs_review|failed",
    "evidence_images": [],
    "mask_candidates": [],
    "role_track_id": "",
    "track_dir": "",
    "quality": {},
    "warnings": []
  },
  "target_asset": {
    "status": "missing|uploaded|generated|confirmed",
    "active_image": "",
    "versions": []
  },
  "usage": {
    "shot_ids": [],
    "time_ranges": []
  },
  "seedance_policy": {
    "send_source_image": false,
    "send_target_image": true
```

关键要求：

3. 没有 target 图时，资产卡显示“待替换”，不能把 source 图伪装成 target。
4. `send_source_image=false` 是默认规则，除非后续专门做调试开关。

## P0 实施步骤

### P0-1：资产展示先分层

改资产卡逻辑：

1. 左侧显示“原片识别证据”：原图、彩色 mask、身份轨 crop、质量警告。
2. 右侧显示“目标替换资产”：用户上传的新角色图、版本、确认状态。
3. 角色、场景、物品分组展示。
4. 明显错误/冲突的证据不进入 ready 状态，只进“待核对”区域。

验收：

1. 用户不会再以为原视频截图就是要替换成的新人物。
2. 同一张证据图分给两个角色时，两个角色都标“身份冲突/待核对”。

### P0-2：自动生成/读取彩色 mask 候选

生成分镜草案或整理资产库后，对关键候选帧自动准备：

1. 原图预览。
2. 彩色 mask 预览。
3. 每个颜色 object 的 bbox、中心点、面积、候选图片。
4. 缓存路径：`assets/identity_candidates/maskcand_*/result.json`。

服务器策略：

1. 如果服务器开着，按用户选择的 SAM3 来源跑。
2. 如果服务器关着，只读缓存；没有缓存就显示“暂无彩色 mask，服务器关闭，不能生成”。
3. 不允许一直显示“生成中”。

验收：

1. 关闭服务器时不阻塞、不扣费、不无限转。
2. 有缓存时能直接展示原图 + 彩色 mask。

### P0-3：同一帧多角色绑定

标注方式改成以“颜色 object”为中心：

1. 用户点击红/蓝/绿等颜色区域。
2. 弹出角色名下拉列表。
3. 选择后保存为：`sam3_object_id + role_id + time + frame + bbox + point`。
4. 同一张图可以保存多个角色绑定。
5. 不限制“当前卡片是哪一个角色”，因为一张画面本来可能有两个人。

UI 文案必须明确：

1. “红色区域属于谁？”
2. “蓝色区域属于谁？”
3. “已绑定：红色=女主，蓝色=黑衣男子。”

验收：

1. 两人同框时，用户能一次把两个人都绑定完。
2. 不再出现“这区域属于哪个人”但看不出指红还是蓝的问题。
3. 点位坐标必须落在图片真实位置，不能漂移到旁边。

### P0-4：跑身份分轨要有明确反馈

接通现有 `/api/storyboard-role-tracks/run`：

1. 按已经绑定的角色 object 跑 SAM3 身份轨。
2. 每个角色生成 `track_summary.json`、mask 序列、identity sheet/crop。
3. 把角色出现时间段写回角色资产和相关 shot。
4. 前端显示 job id、进度、成功/失败数量、失败原因。

验收：

1. 点击“跑身份分轨”必须马上显示任务已开始或无法开始原因。
2. 服务器关闭时按钮显示不可用原因，不假装运行。
3. 分轨成功后，资产卡的原片证据更新为 identity sheet，而不是单张误选脸。

### P0-5：资产库自动体检后再展示

整理资产库时自动做基础体检：

1. 角色没有身份证据：待核对。
2. 两个角色使用同一张 source 图：身份冲突。
3. source 人物占比异常：待核对。
4. 场景图人物占比过高：降级为镜头参考，不当干净场景。
5. 物品/场景语义和画面冲突：待核对。

验收：

1. 错误资产不会以“可用资产”的样子展示。
2. 体检结果能自动改变资产状态，而不是只生成一段文字让用户自己猜。

## P1：场景/物品多帧择优与提纯

P1 的核心是：画面为主，语义只是线索。不能因为反推文本说“床”就拿一张人物近景当床，也不能因为文本说“粗链条”就把含有其它物件的混乱帧当成物品资产。

### P1-1：同资产多候选帧

每个场景/物品资产不再只存一张 `source_image`，而是存候选列表：

```json
{
  "candidates": [
    {
      "image": "",
      "time": 12.4,
      "shot_id": "shot_003",
      "visual_score": 0.82,
      "semantic_score": 0.61,
      "person_area_ratio": 0.08,
      "object_mask_area_ratio": 0.24,
      "warnings": []
    }
  ],
  "selected_candidate": ""
}
```

择优规则：

1. 场景优先选空镜、远景、人物占比低、稳定清晰的帧。
2. 物品优先选目标物体占比高、遮挡少、边缘完整的帧。
3. 人物近景不能当干净背景，只能标为 `shot_reference`。
4. 多个语义相近资产先合并候选，例如“粗链条”和“粗铁链”应进入同一个物品资产。

### P1-2：检测/分割/提纯模型

服务器已有 SAM3 和 ComfyUI 通路，当前已装 LaMa `big-lama.pt`，GroundingDINO 模型文件也已下载但节点暴露曾返回 `{}`，需要后续确认节点名称/依赖是否完整。

推荐模型职责：

1. SAM3：人物/区域分割、颜色 mask、人物轨迹、候选区域。
2. GroundingDINO：按文本找物品或场景元素，例如 chain、bed、door、car。
3. SAM/SAM3：把 GroundingDINO 找到的框细分成 mask。
4. LaMa：从场景图里擦除人物，生成 clean background 候选。
5. YOLO/人脸检测：粗人物数、人物占比、误用人物近景报警。

P1 代码策略：

1. 服务器开时跑检测和提纯。
2. 服务器关时只读缓存，不阻塞。
3. 每个资产保留原帧、mask、提纯图、评分和警告。
4. 体检结果直接影响展示：可用、待核对、隔离、镜头参考。

### P1 验收

1. “床”资产不会自动选到非床帧。
2. “粗链条/粗铁链”会合并成一个物品资产或明确提示重复。
3. 场景资产默认优先干净背景，人物占比高时自动降级。
4. 资产卡能看到原图、mask、提纯图、候选评分，并支持单个资产重新提纯。

## P2：分镜管理与 Seedance 准备层

P2 的核心是把竞品里好用的“横向时间轴 + 当前镜头编辑 + 资产引用”做进 Mode2，但保持代码独立。

### P2-1：横向时间轴

分镜列表不再上下堆叠，而是底部横向时间轴：

1. 每个 shot 是 4-15 秒窗口。
2. 时间轴展示镜头缩略图、起止时间、人物数、主要角色、状态。
3. 点击 shot 后，中间区域切到当前镜头详情。
4. 左右支持原视频/生成视频对比。

### P2-2：当前镜头编辑区

当前镜头显示：

1. 原视频片段预览。
2. 关键帧/构图参考。
3. 当前镜头提示词。
4. 当前镜头关联资产：角色、场景、物品。
5. 生成状态：未生成、生成中、已生成、失败、需重跑。

### P2-3：Seedance 策略

Seedance2.0 当前策略：

1. 原视频片段只当动作、镜头、节奏骨架。
2. 目标角色图、目标场景图、目标物品图才是生成参考资产。
3. 彩色 mask/SAM3 分轨不传给 Seedance，只用于我们内部确定“这个 shot 里应该绑定谁”。
4. 提示词尽量短而明确，不写 Seedance 不吃的复杂 mask 逻辑。

模板方向：

```text
Use the source video as motion and camera reference. Keep the same action, facial expression, blocking, camera angle, distance, pacing, and interaction.
Replace the original characters with the provided target character references. Keep the story unchanged. Change the language/dialogue to English and restyle the setting as realistic European drama.
Avoid extra people, face swapping between characters, duplicated bodies, distorted limbs, text, subtitles, watermark.
```

### P2 验收

1. 用户可以像剪辑软件一样横向选择 shot。
2. 每个 shot 只显示相关资产，不把全资产库都塞进来。
3. 每个 shot 的生成参数和资产引用可复制、可编辑、可重跑。
4. 旧模式代码可继续保留，但新模式生成链路不依赖旧模式。

## 本轮不做的事

1. 不把彩色 mask 传给 Seedance 当生成控制。
2. 不做最终 Seedance 批量生成，只准备 shot 级策略和资产引用。
3. 服务器已关闭时不做远程 SAM3/GroundingDINO/LaMa 实测。
4. 不把 Mode1 转绘流程接进 Mode2。
5. 不因为旧项目参考触发昂贵重跑；旧项目默认只读缓存。

2. P0：修彩色 mask 候选状态，缓存优先，服务器关闭时清楚失败。
3. P0：改身份绑定 UI，支持同一张 mask 上多个颜色 object 分别绑定角色。
4. P0：接通跑分轨按钮反馈，能显示 job、进度、失败原因。
5. P0：让资产体检结果直接改变资产状态。
6. P1：给场景/物品资产补多候选、评分、人物占比警告和提纯状态字段。
7. P1：预留单资产重新提纯入口，服务器关时只返回待跑/缓存缺失。
8. P2：补横向时间轴和当前镜头资产关系字段，先保证数据和界面能承载。
9. 本地做语法检查和旧项目缓存加载测试；远程 SAM3 等服务器开启后再测。

## P0 完成标准

P0 完成后，打开 Mode2 资产库应该能做到：

1. 看见角色/场景分组。
2. 每个角色清楚分成原片证据和目标替换图。
3. 两人同框时，可以用彩色 mask 一次绑定两个人。
4. 绑定后可以跑身份分轨，并看到明确进度。
5. 不可信资产不会混进可用资产区。
6. Seedance 草稿只拿目标替换图，不误拿原片证据图。

## 服务器开启后的测试清单

1. 选择旧项目参考、禁用强制重跑，确认不会重新扣费。
2. 选择一个短视频，开启 SAM3 服务器，点击“分析原视频”，确认资产库和分镜同时生成。
3. 打开彩色区域绑定，确认原图和彩色 mask 同时显示，同一帧能绑定多人。
4. 绑定后跑身份分轨，确认角色证据图和 shot 角色绑定被刷新。
5. 对场景资产点“重新提纯”，确认人物占比高的图被 LaMa clean background 或待核对状态替换。
6. 进入分镜页，确认 4-15 秒窗口、相关资产、Seedance 策略字段都能看到。
